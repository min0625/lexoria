# 英文填字遊戲（Wordscapes-like）設計文件

> 第一階段：純 HTML/CSS/JS Web 版，之後嵌入 iOS / Android。
> 差異化賣點：已找到的單字可查釋義與發音——玩遊戲順便學英文。

**章節導覽**：玩法（§1）→ 技術選型（§2–4）→ 內容與資料（§5–6）→ 遊戲設計（§7–8）→ 工程實務（§9–12）→ 平台嵌入（§13）→ 授權（§14）→ 計畫與驗收（§15–19）。
**配套文件**：[畫面清單與 UI 流程](docs/word-puzzle-ui-screens.md)——畫面線框、導覽流程、overlay 疊層、UI 文案表、色票草案。

## 1. 核心玩法

- 畫面下方是一個「字母轉盤」（letter wheel），有 5–7 個字母。
- 玩家用手指滑過字母連成單字，畫面上方的填字格（crossword grid）逐格填入。
- 找出全部目標單字即過關；額外拼出的合法單字算 bonus word，給金幣。
- 金幣可買提示（揭示某一格字母）。
- 沒有計時、沒有失敗狀態——卡住只是還沒找到，這是解謎不是動作遊戲。

### 容易被忽略的規則細節

- 最短單字長度為 3（1–2 字母不進判定流程）。
- 轉盤字母**可以重複**（如 APPLE 有兩個 P）：選取狀態要綁在「按鈕實例」上而不是字母值上，已選過的按鈕不能再選，判定時也要比對字母數量而非只比對集合。
- 已找到的目標字再拼一次 → 只提示「已找到」，不重複播放飛入動畫。
- 已找到的 bonus 字再拼一次 → 不重複給金幣（`foundBonusWords` 按關卡記錄）。
- 轉盤中央放「洗牌」按鈕（Wordscapes 標配）：重排字母位置幫玩家看出新單字，免費、不限次數。保證至少換一個位置——洗回原樣看起來像按了沒反應。
- 手勢進行中按到洗牌 → 不生效（字母位置移動會讓進行中的手勢座標錯亂）。

## 2. 架構總覽

```
┌─────────────────────────────┐
│  Native Shell (iOS/Android)  │  ← 第二階段
│  ┌───────────────────────┐  │
│  │  WebView (遊戲本體)     │  │
│  │  HTML + CSS + JS       │  │
│  │  levels.json (關卡)     │  │
│  │  localStorage (進度)    │  │
│  └───────────↕────────────┘  │
│     JS Bridge (廣告/IAP/存檔) │
└─────────────────────────────┘
```

**建議：第一階段完全不需要框架與後端。**

- 純 Vanilla JS + ES Modules 就夠了。遊戲畫面只有兩個元件（格子、轉盤），引入 React/Vue 只會增加打包複雜度和 WebView 載入時間。
- 不需要打包工具（起步時）：直接 `<script type="module">`，等真的需要再加 Vite。
- 不需要後端：關卡是靜態 JSON，進度存 localStorage。
- 全部資源本地打包 → 天生離線可玩，這對嵌入 App 是硬需求（不能讓玩家在飛機上開不了遊戲）。

## 3. 渲染技術選擇

| 部分 | 建議 | 理由 |
|------|------|------|
| 填字格 | CSS Grid + DOM | 格子是靜態排版，DOM 最省事，動畫用 CSS transition |
| 字母轉盤 | DOM 按鈕 + 一層 SVG 疊在上面畫連線 | 手勢命中判斷用座標計算，連線軌跡用 SVG polyline 即時更新 |
| 粒子/慶祝特效 | CSS animation，之後不夠再上 Canvas | 過關撒花不值得引入遊戲引擎 |

**不建議用 Phaser/Pixi 等遊戲引擎**：這個玩法沒有連續的遊戲迴圈（game loop），本質是回合制 UI，引擎只會增加 1–2MB 載入量。

## 4. 觸控與手勢（最容易踩坑的部分）

字母轉盤的滑動連字是整個遊戲的手感核心：

- 用 **Pointer Events**（`pointerdown` / `pointermove` / `pointerup`），一套 API 同時涵蓋滑鼠與觸控。
- `pointerdown` 時對轉盤容器 `setPointerCapture`：手指滑出轉盤範圍、甚至在畫面外放開，`pointerup` 仍會送到轉盤，手勢才能正確收尾（不做這個會出現「字母卡在選取狀態」的鬼 bug）。
- 轉盤容器設 `touch-action: none`，否則滑動會觸發頁面捲動/下拉刷新。
- 全域設 `user-select: none` 與 `-webkit-touch-callout: none`，避免長按選字、iOS 放大鏡。
- 命中判斷**不要**依賴 `elementFromPoint`，改用「手指座標與每個字母圓心的距離 < 半徑」自己算，命中範圍可以調大一點（實際按鈕的 1.2 倍），手感會好很多。
- 支援「滑回上一個字母 = 取消最後一個字母」（Wordscapes 的標準行為）。
- viewport 設定：`<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">`，並用 `env(safe-area-inset-*)` 處理瀏海與 home indicator。

## 5. 關卡資料設計

關卡在**建置期離線產生**，不要在執行期算：

```json
{
  "id": 42,
  "letters": ["A", "E", "L", "P", "N"],
  "words": [
    { "word": "PLANE", "row": 0, "col": 2, "dir": "across" },
    { "word": "LEAP",  "row": 0, "col": 2, "dir": "down" },
    { "word": "PANE",  "row": 3, "col": 0, "dir": "across" }
  ],
  "bonus": ["PEAL", "NAPE", "PLEA"]
}
```

產生器（一次性 Node/Go/Python 腳本）的流程：

1. 選一個 3–7 字母的基底單字（頻率表取常用字，冷僻字會毀掉體驗；字母數依難度帶而定，見步驟 5）。
2. 枚舉其字母的所有子集合排列，對照字典找出所有合法子單字。
3. 挑 2–8 個「常見」的當目標單字（數量依難度帶而定），其餘進 bonus 清單。
4. 用簡單的回溯演算法把目標單字排進交叉格局。
5. 難度曲線：開頭 1–2 關用 3 字母（配合 §7 的教學關），前期 3–4 字母、單字都很常見；後期字母多、含較少見單字。
6. **驗證器（同一支腳本的最後一步）**：每個交叉格的字母一致、每個目標字都能由轉盤字母組成（含重複字母的數量）、格局是連通的、同一關無重複單字——任何一關驗證失敗就讓整批建置失敗。產出後人工抽玩前 10 關確認手感。

### 產生器演算法細節

**步驟 2（枚舉子單字）——不要枚舉排列。** 7 個字母的全排列子集是幾十萬次字典查詢；正確的懶做法是 **alphagram 索引**：

1. 把整本字典預處理成 `排序後字母 → [單字]` 的 map（`"AELNP" → ["PANEL", "PLANE"]`）——這就是本節末尾 bonus 字典的同一份資料結構，產生器和遊戲執行期共用，只寫一次。
2. 基底單字的字母組合最多 2⁷ = 128 個子集（multiset 子集，重複字母按數量算），每個子集排序後查一次 map。128 次查詢，毫秒級完成。

**步驟 4（交叉格局）——回溯擺放：**

```
place(words):                          # words 依長度遞減排序
  第一個字橫放在 (0,0)
  對其餘每個字 w：
    對每個已擺上的字 p、每對相同字母 (w[i] == p[j])：
      嘗試把 w 垂直於 p 擺放，交叉點對齊該字母
      合法 = 每格「空白或字母相同」
           且 w 頭尾外側一格是空的
           且 w 每個非交叉格的左右（或上下）鄰格是空的   # 避免產生意外的相鄰字
      合法 → 擺上，遞迴下一個字
    全部位置都失敗 → 回溯上一個字換位置
  全部字都擺上 → 成功
```

- 一關只有 4–8 個字，回溯的搜尋空間極小，暴力就是最佳解，不需要任何剪枝技巧。
- 「非交叉格的鄰格必須是空的」這條規則最容易漏——漏了會在格上拼出不在題目裡的相鄰字串，玩家看得到但點不了。
- 徹底擺不進去的字（罕見）→ 丟回 bonus 清單，從候選裡遞補一個。

**挑最好看的一版：** 同一組字用不同隨機順序跑 20 次，計分 = 交叉點數多、bounding box 小、長寬比接近直式螢幕，留最高分。「多產幾版挑最好」比設計聰明演算法便宜得多。

**可重現性：** 亂數種子固定為關卡 id。重跑產生器輸出完全相同，關卡才能進版控、diff 才有意義。

**字典建議**：目標單字用常用字頻表過濾；bonus 判定用大字典（建議 ENABLE，public domain）。注意授權陷阱：TWL / SOWPODS 是 Scrabble 主辦方的專有字表、COCA 頻率表商用要付費，都不要用——各資料源授權與替代品整理在 §14。兩份分開——「玩家想得到的字被判無效」是這類遊戲最大的差評來源。

Bonus 字典幾十萬字，全部塞進關卡 JSON 不划算 → 打包成一份以字母排序 key 的 JSON（`"AELNP": ["PANEL","PLANE",...]`）或前綴樹，查詢 O(1)。壓縮後約幾百 KB，可接受。啟動時 `fetch` 非同步載入一次、常駐記憶體；`levels.json` 很小，隨頁面直接載入即可。

## 6. 單字解釋與發音（查詞功能）

這功能其實就是「多一份資料 + 一個按鈕」，不需要新系統。

### 6.1 觸發方式

- 玩家**已找到**的字才能查（未找到的字不能查，否則變相洩題）。
- 點擊/長按已填入的目標字格 → 彈出小卡片：釋義文字 + 一個喇叭圖示。
- 喇叭按下才發音，不自動念——滑進格子就自動出聲，連續拼字時會很吵。
- 例外：**答對目標字的當下自動念一次**，此時發音**取代**命中音效，若該字同時觸發過關，過關音效也不播、由過關卡片承擔慶祝感（實測疊播或緊接人聲都很突兀；無英文語音時退回原音效）——bonus 字不念，連續拼 bonus 會很吵。自動發音受設定的音效開關控制；查詞卡的喇叭鈕是玩家明確點擊，不受此開關影響。

### 6.2 發音：用 Web Speech API，不要打包音檔

```js
function speak(word) {
  const u = new SpeechSynthesisUtterance(word);
  u.lang = 'en-US';
  speechSynthesis.speak(u);
}
```

- 免費、免打包音檔、免網路——`speechSynthesis` 是瀏覽器內建 API，直接呼叫系統 TTS。
- iOS Safari / WKWebView、Android WebView 皆支援；Phase 2 包進 Capacitor 後一樣能用（走系統 TTS，不需要額外的 native plugin）。
- **不建議**改用預錄真人發音音檔或線上發音 API：幾千個單字的音檔要嘛占容量、要嘛破壞第一階段「全部資源本地打包、天生離線可玩」的硬需求（見 §2），TTS 品質對這個遊戲完全夠用，真有玩家反應機器音很怪再評估。
- 呼叫前用 `speechSynthesis.getVoices()` 確認至少有一個英文語音；沒有就隱藏喇叭按鈕（少數精簡版 Android WebView 可能沒裝語音包），不要讓按鈕點下去沒反應。

### 6.3 釋義：離線打包一份精簡資料，不要打 API

現有的 bonus 字典（§5）只有單字表、沒有釋義，需要另外一份資料：

- 英文釋義用 **WordNet**（Princeton；WordNet License，免費可商用但需隨附授權聲明，見 §14）：每個字取第一個常用義的一句短釋義即可，不做詞性標註、多義選擇、例句這類進階資訊。
- 繁體中文釋義用 **ECDICT**（MIT，見 §14）：取 translation 欄第一行（同樣是一句短釋義的原則），原始資料為簡體，建置期以 **OpenCC**（s2twp）轉為台灣正體。查詞卡以中文為主、英文釋義為輔。
- 只需要涵蓋關卡實際用到的目標字（本來就是常用字頻表過濾出來的少量字），不必打包整本字典——產生器（§5 步驟 3 挑目標字時）順便查一次塞進 JSON。
- 直接內嵌進 `levels.json`（每關字少，內嵌最省事，不必再建索引檔）：

  ```json
  { "word": "PLANE", "row": 0, "col": 2, "dir": "across", "def": "a flat or level surface", "zh": "n. 平面, 飛機" }
  ```

- **不建議**呼叫線上字典 API（如 dictionaryapi.dev）：查詞功能沒有正當理由讓「全部本地打包」的原則破例，而且免費字典 API 通常有 rate limit，玩家連續查字會被擋。
- Bonus 字先只做發音（TTS 不需要額外資料，任何字都能念），不做釋義文字——見 §16 YAGNI。

## 7. 畫面與 UX 要點

- **直式鎖定**、單手可玩：轉盤在下三分之一（拇指區），格子在上方。
- 格子尺寸用 `min(vw, vh)` 相對單位計算，適配各種螢幕比例（包含平板）。
- 即時回饋：滑字時上方顯示目前拼出的字母串；放開時——
  - 命中目標字 → 字母飛入格子動畫；
  - 命中 bonus → 金幣 +N 動畫；
  - 重複的字 → 短暫「已找過」提示；
  - 無效字 → 字串抖動後消失。
- Haptic（震動）在命中時給一下，嵌入 App 後體感差很多，值得做。
- 深色模式：用 CSS custom properties 定義色票，跟隨 `prefers-color-scheme`。
- Web 版鎖不了螢幕方向：橫式時蓋一層「請轉直畫面」遮罩即可，不要為橫式做第二套排版。
- 教學：第 1 關用 3 字母 + 一次性的滑動手勢示意（手指軌跡 overlay），玩家完成第一個字就收掉、不再出現；不做多步驟教學精靈。
- 關卡選擇：線性解鎖（最遠玩到 `currentLevel`）；已完成關卡可重玩，但不重複給金幣（獎勵已按關卡記錄）。
- 已找到的目標字可點擊查看釋義與發音（見 §6），卡片位置貼著被點的格子彈出，點卡片外任意處關閉。
- 音效只要 5 個短音：選字 tick、命中目標字、無效字、金幣、過關。用 Kenney / freesound 的 CC0 素材，Web Audio 播放，不放背景音樂。
- `prefers-reduced-motion` 開啟時跳過飛入/撒花動畫，直接顯示結果。
- 設定入口（齒輪按鈕）：音效、震動開關，存進存檔的 `settings`。
- 關卡全數破完後顯示「更多關卡即將推出」畫面——50 關被玩完的速度比想像中快，別讓玩家點下一關時當掉。

## 8. 金幣與提示經濟（初版數值，之後照數據調）

| 事件 | 金幣 |
|------|------|
| 拼出一個 bonus 字 | +2 |
| 過關 | +10 |
| 使用提示（揭示一格） | −25 |
| 新玩家初始 | 50（夠用兩次提示，教學關順便教用法） |

- 數值全部集中成一個常數物件（放 `game.js` 頂部即可），不要散在程式各處。
- 原則：正常遊玩下每 3–4 關能存到一次提示——卡關有救，但不會氾濫到無腦點提示。

### 提示的具體行為（邊界情況最多的功能）

- 提示 = 隨機揭示一格**未填**的格子，揭示後直接顯示字母並鎖定。不做「指定格子揭示」的進階提示——一種提示先驗證夠不夠用。
- 一個目標字的所有格子都被填滿（不論來自拼字還是提示）→ 即算找到：照常播動畫、計入勝利判定、轉盤拼它時提示「已找到」。
- 拼出「部分格子已被提示揭示」的字 → 照常整字填入，算玩家找到。
- 金幣不足 → 提示按鈕抖動並顯示價格，不彈任何購買視窗（IAP 是 Phase 3 的事）。

## 9. 狀態與進度儲存

- 第一階段：`localStorage` 存 `{ version: 1, currentLevel, coins, foundBonusWords, levelState: { foundWords, revealedCells }, settings: { sound, haptic } }`，一個 key、一個 JSON，夠了。
- `levelState` 是**進行中關卡**的進度（已找到的目標字、提示揭示過的格子）：玩家中途關掉頁面不該歸零——提示是花金幣買的，丟了等於扣錢沒拿到東西。過關進下一關時清空。
- `version` 欄位第一天就放：日後存檔格式變更才有辦法寫遷移；讀到壞掉或無法辨識的資料時，重置成初始存檔（Phase 1 沒有付費資產，重置的代價可以接受）。
- **重要**：把存檔讀寫包成一個小模組（`storage.js`，約 20 行），第二階段嵌入 App 時只要換掉這一個模組改走 native bridge（§13），遊戲邏輯完全不動。這是唯一值得預留的抽象層。

## 10. 模組介面與事件流

`main.js` 負責接線；模組之間不互相 import，也不需要 event bus——資料流只有一條，直接回呼就夠。

```
wheel.js ──手勢收尾(word)──▶ game.submit(word) ──結果物件──▶ main.js 分派給 grid / HUD / 特效
```

`game.js` 對外只有兩個動作，回傳的結果物件是唯一需要事先約好的介面：

```js
submit(word) →
  { type: 'target',    word, cells, won }   // cells: 要填入的格子座標；won: 這一下是否過關
  { type: 'bonus',     word, coins }        // coins: 本次獲得的金幣數
  { type: 'duplicate', word }               // 目標字或 bonus 字已找過
  { type: 'invalid',   word }               // 不在任何字典

useHint() →
  { ok: true, cell, letter, completedWords, won }  // 揭示哪一格；順便補滿的字與是否過關（§8 邊界）
  { ok: false, reason: 'no-coins' }
```

判定順序固定「先目標、後 bonus」（產生器保證兩份清單不重疊）：

```
submit(word):
  是本關目標字 → 已找過？duplicate：填格、判勝利 → target
  在 bonus 字典 → 已在 foundBonusWords？duplicate：加金幣 → bonus
  其餘 → invalid
```

長度 < 3 的字串由 `wheel.js` 直接吞掉，不會進到 `submit`（見 §1 規則細節）。

格子狀態只有三種、單向轉移。`revealed` 與 `filled` 畫面上長一樣，分開記只是為了日後統計提示使用率（§15 Phase 3）：

```
empty ──玩家拼出──▶ filled
empty ──提示揭示──▶ revealed
```

## 12. 測試策略

- **手勢與判定邏輯寫成純函式**（座標 → 命中的字母、拼出的字串 → 目標/bonus/重複/無效），與 DOM 分離，直接用 `bun test` 跑單元測試——這是唯一值得自動化的部分。
- 關卡資料品質靠產生器內建的驗證器把關（見 §5），遊戲端不再重複驗證。
- UI、動畫、手感用真機手動測：iOS Safari + 一台中低階 Android 是最低測試矩陣；桌面 Chrome 的觸控模擬不能取代真機。手測以 §17 的驗收清單為準。
- 開發期在桌面支援鍵盤輸入（按字母鍵選字、Backspace 取消最後一個、Enter 送出）——不是玩家功能，是讓開發迭代不必每次用滑鼠畫圈，改邏輯後跑回歸快很多。

## 13. 嵌入 iOS / Android 的策略

### 選項比較

| 方案 | 說明 | 建議 |
|------|------|------|
| **Capacitor** | 官方外掛涵蓋儲存、IAP、廣告、haptics；社群成熟 | ✅ 首選 |
| 自寫 WKWebView / WebView shell | 最輕，但 bridge、生命週期、鍵盤、儲存都要自己來 | 除非有特殊需求 |
| Cordova | 老舊，生態萎縮 | ❌ |
| React Native WebView | 只為了包 WebView 而引入 RN 不划算 | ❌ |

### Bridge 介面先想好（但先不要實作）

第一階段只要定義好呼叫點，全部給 web fallback：

```js
// bridge.js — 第一階段全是 web 實作，第二階段逐個換成 native
export const bridge = {
  save(data)      { localStorage.setItem('save', JSON.stringify(data)); },
  load()          { return JSON.parse(localStorage.getItem('save') ?? 'null'); },
  haptic(type)    { navigator.vibrate?.(10); },
  showAd()        { return Promise.resolve(); },   // 之後接 AdMob
  buy(productId)  { return Promise.reject(); },     // 之後接 IAP
};
```

遊戲程式碼只准 import `bridge`，不准直接碰 `localStorage` 或 native API。

### WebView 相關注意事項

- iOS WKWebView 的 localStorage 在某些情境（App 被系統清理）可能丟失 → 嵌入後改由 native 端存檔（Capacitor Preferences / filesystem）。
- 音效：行動瀏覽器要求「首次使用者手勢後」才能播音 → 在第一次 `pointerdown` 時初始化 Web Audio context。
- 禁用 WebView 的縮放與長按選單（Capacitor 預設已處理大半）。
- 效能：動畫只用 `transform` / `opacity`（compositor-only），避免觸發 layout；中低階 Android 機是效能底線。
- **上架風險（Apple 審查準則 4.2「最低功能性」）**：純 WebView 包殼的 App 有被拒的前例。緩解：完全離線可玩（本來就是 §2 的硬需求）、接上 native haptics、完整的啟動畫面與 App icon、UI 不露出任何瀏覽器痕跡。Capacitor 遊戲過審前例很多，但第一次送審的時程要預留被拒一輪的緩衝。

## 15. 開發階段規劃

### Phase 1 — Web MVP（目標：手機瀏覽器可玩）
1. 核心玩法：轉盤手勢、格子填入、勝利判定（手刻 3–5 個關卡即可開始）。
2. 關卡產生器腳本 + 產出前 50 關（順便產出目標字釋義，見 §6.3）。
3. bonus 字判定、金幣、提示功能。
4. 查詞與發音（§6）：格子加點擊查詞卡片，喇叭按鈕接 Web Speech API。
5. 進度儲存、關卡選擇畫面。
6. 音效與基本動畫。
7. 部署到靜態託管（GitHub Pages / Cloudflare Pages 皆免費），用真機測試、發連結找人試玩。

### Phase 2 — 嵌入 App
1. Capacitor 包殼，iOS / Android 各出一個 build。
2. 存檔改走 native（換掉 `bridge.save/load` 實作）。
3. Haptics、啟動畫面、App icon。
4. 上架 TestFlight / 內部測試。

### Phase 3 —（有市場驗證後才做）
- 廣告（激勵影片換提示）、IAP（去廣告、金幣包）。
- 輕量遊玩統計（過關率、提示使用率、流失在哪一關）——§8 的數值要靠這個調，接廣告 SDK 時順便接。
- 每日挑戰、雲端存檔、成就。
- 更多語言的關卡包。
- Bonus 字的釋義文字（見 §16 YAGNI）——有數據顯示玩家真的常查目標字釋義，再評估要不要擴大到 bonus 字。

## 16. 刻意不做的事（YAGNI）

- ❌ 後端 / 帳號系統 —— 單機離線遊戲，等有雲端存檔需求再說。
- ❌ 前端框架 —— 兩個元件的 UI 用不到。
- ❌ 遊戲引擎 —— 沒有 game loop 的回合制 UI。
- ❌ 即時對戰 / 排行榜 —— Phase 3 以後的事。
- ❌ 執行期產生關卡 —— 離線產生品質可控、可人工審核。
- ❌ PWA / Service Worker —— Phase 2 用 Capacitor 打包後天生離線；Web 版只是測試通路，不值得維護快取失效邏輯。
- ❌ 多種提示類型（指定格、開頭字母、火箭）—— 先驗證一種隨機揭示夠不夠。
- ❌ Bonus 字釋義、多義項/詞性/例句、發音腔調選擇（美式/英式）—— 目標字的單一短釋義 + 系統預設 TTS 就夠，見 §6。
- ❌ 線上字典 / 發音 API —— 破壞離線可玩的硬需求，Web Speech API + 本地打包的 WordNet 釋義已經足夠（§6.2、§6.3）。

## 18. 專案名稱建議

命名時避開「Word + 常見名詞」這個已經被用爛的套路——Wordscapes、Word Cookies、Words of Wonders、Wordloom、Wordnest、Wordgrove、Lexitrail、Vocabloom、WordWyrm 都是實際存在的產品，撞名風險高。以下候選已用網路搜尋做過初步排除（非正式商標查詢）：

| 名稱 | 檢查結果 | 特點 |
|------|---------|------|
| **Lexoria**（推薦） | 只查到一個無關的 Twitch/TikTok 個人帳號，沒有 App 或品牌撞名 | lex（字彙）+ -oria（地域感字尾），呼應 §6 查詞典/發音這個差異化賣點 |
| Wordelia | 沒查到任何撞名 | word + -elia 字尾，親切好記，適合做吉祥物角色 |
| Wordune | 只有拼法接近的「Word Tunes」（無關的音樂類單字遊戲） | word + dune，呼應洗牌機制的字母重新排列 |
| Verbolo | 已被一個無關類別的影片編輯 App 用掉 | 風險中等，可用但非首選 |

**推薦 Lexoria**：跟市面一堆 "Word+X" 命名的字謎遊戲拉開品牌距離，"Lex-" 字根也呼應查詞典/發音功能——這是這款遊戲跟一般填字遊戲最大的差異賣點。

⚠️ 上述只是網路搜尋的初步排除，正式定案前應查 App Store / Google Play 名稱搜尋，以及台灣 TIPO 商標資料庫，確認沒有註冊商標衝突。

## 19. 開工前待決事項

寫第一行程式前要拍板的事——都不影響架構，但影響資料與素材的準備方向：

1. **UI 語言** —— 介面文案用英文還是繁中？WordNet 釋義只有英文；若定位是台灣市場「玩遊戲順便學英文」，繁中 UI + 英文釋義是合理組合（中文釋義要另找資料源，工程量大增，Phase 1 不建議）。文案總量很小（十幾條字串），集中成一個 `strings.js` 物件即可，之後換語言只改一個檔——但不要做 i18n 框架。繁中初稿已列在 UI 文件 §6。
2. **名稱定案** —— 照 §18 做 App Store / Google Play / TIPO 查冊；名稱定了才能建 repo、買網域。
3. **視覺風格** —— 開工前定一次色票（CSS custom properties，深淺兩套，§7）與標題字體，避免邊開發邊調設計。可直接開工的草案已列在 UI 文件 §7。
4. **字頻表選擇** —— 見 §14 授權表；這是產生器（§5）第一步的輸入，不定案產生器動不了。
